# 로컬 코드리뷰

커밋 전 변경사항 또는 현재 브랜치의 누적 변경을 리뷰한다. PR로 올라가기 전에
스스로 점검하는 용도이므로, 리뷰어에게 지적받을 만한 것을 미리 잡는 것이 목표다.
pr-review와 같은 프로젝트 메모리를 공유한다 — PR 리뷰에서 발견된 패턴이
로컬 리뷰에도 반영되고, 그 반대도 마찬가지다.

## 1단계: 메모리 준비 및 로드

```bash
RTK_SKIP=1 bash ~/.claude/skills/review-memory/scripts/setup_memory.sh <프로젝트경로>
```

그 다음 반드시 읽는다:
- `.review-memory/context.md` — 사용자가 미리 등록한 배경/중점 사항/팀 규칙. 리뷰 우선순위에 직접 반영한다.
- `.review-memory/conventions.md` — 반복 패턴/컨벤션
- `.review-memory/reviews/` 의 최근 2~3개 — 최근 리뷰 맥락

사용자가 리뷰 요청과 함께 배경 정보를 줬다면 이번 리뷰에 반영하고, "앞으로/항상" 같은
지속 의도가 보이면 context.md에 등록한다 (review-memory 스킬 참조).

> 환경 주의: 이 머신에서는 rtk 훅이 셸 명령을 재작성한다. 이 스킬의 모든 셸 명령
> (git/gh/스크립트)에는 `RTK_SKIP=1` 접두어를 붙여 재작성을 우회한다 — 훅 래퍼가
> 이 마커를 인식해 원본 명령을 그대로 실행한다. 그래도 `_safe_eval` 오류가 나면
> 해당 명령을 절대경로로 재시도하고(예: `/usr/bin/git`, `/opt/homebrew/bin/gh`, `/bin/ls`),
> 파일 읽기/쓰기는 Read/Write 도구를 사용한다.

## 2단계: 리뷰 범위 결정

```bash
RTK_SKIP=1 git status
RTK_SKIP=1 git diff HEAD          # 미커밋 변경 (staged + unstaged)
```

- 미커밋 변경이 있으면 그것이 기본 대상이다.
- 미커밋 변경이 없거나 사용자가 "브랜치 전체"를 원하면: `git diff <기본브랜치>...HEAD` (기본 브랜치는 `git remote show origin` 또는 main/master 존재 여부로 판단)
- 어느 쪽인지 애매하면 두 범위의 크기를 보여주고 확인한다.

diff에 새 파일이 있으면 전체를 읽고, 수정된 파일은 변경 주변 맥락과 호출처를 확인한다.
**diff 바깥의 영향**(수정된 함수의 다른 호출처, 깨질 수 있는 사용처)을 확인하는 것이 핵심이다.

## 3단계: 리뷰

PR 리뷰와 같은 관점이지만, 커밋 전이므로 더 일찍 잡을수록 싼 것들에 비중을 둔다.
context.md의 중점 사항이 있으면 가장 먼저 체크한다:

1. **정확성** — 버그, 엣지 케이스, 잘못된 로직
2. **프로젝트 컨벤션** — conventions.md 패턴 + 기존 코드 스타일과의 일관성
3. **빠뜨린 것** — 디버그 코드/console.log 잔재, 주석 처리된 코드, 임시 하드코딩, 같이 수정했어야 할 파일
4. **테스트** — 새 로직에 테스트가 있는지
5. **커밋 위생** — 의도와 무관한 변경이 섞여 있지 않은지

확신 있는 소수의 발견이 스타일 지적 나열보다 가치 있다.

## 4단계: 결과 출력

읽는 사람이 (1) 문제가 된 **실제 코드**를 바로 보고, (2) `파일경로:라인`을 **클릭해서 에디터로 이동**할 수 있어야 한다. 발견 사항은 한 줄 나열이 아니라 항목별 블록으로 쓴다. ALWAYS 이 구조를 사용한다:

````markdown
# 로컬 리뷰: <브랜치명> (<범위: 미커밋 변경 | main 대비>)

## 요약
(변경이 무엇을 하는지 + 전체 평가 2~3문장)

## 발견 사항

### 🔴 커밋 전에 수정

#### 1. <문제를 한 줄로 요약한 제목>
📍 `상대경로/파일.js:시작-끝`

```<언어>
<해당 코드 영역 — 문제가 보이는 최소한의 줄들>
```

**문제**: 왜 문제인지 1~3문장.
**수정 제안**:
```<언어>
<고친 코드>
```

### 🟡 권장
(같은 블록 구조)

### 💬 확인 필요
(같은 블록 구조 — 질문 형태로)
````

- 위치는 `파일경로:라인` 형식을 지킨다 — 터미널/IDE에서 클릭하면 해당 위치로 바로 이동한다.
- 코드 블록은 문제가 보이는 최소 범위만. 맥락이 필요하면 주변 2~3줄까지.
- 수정 제안 코드는 그대로 적용 가능한 완성형으로 쓴다.

수정은 사용자가 요청할 때만 적용한다. 기본 동작은 발견 사항 보고다.

## 5단계: 메모리 저장

1. `.review-memory/reviews/<YYYY-MM-DD>-local-<브랜치명>.md` 에 리뷰 결과 저장 (날짜/브랜치/범위를 헤더로)
2. **conventions.md 업데이트 판단** — pr-review와 같은 기준:
   - 반복해서 나오는 이슈 → 기존 항목에 횟수/날짜 추가
   - 프로젝트 고유의 새 패턴 → 새 항목 추가
   - 일회성 실수는 기록하지 않는다

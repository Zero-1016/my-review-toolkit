# PR 리뷰

GitHub PR을 리뷰한다. 핵심 차별점은 **프로젝트 메모리**다: 리뷰 전에 이 프로젝트에서
과거에 발견된 패턴을 읽고, 리뷰 후에 새로 알게 된 것을 기록한다. 리뷰를 거듭할수록
이 프로젝트에 특화된 리뷰어가 되는 것이 목표다.

## 1단계: 메모리 준비 및 로드

```bash
bash ~/.claude/skills/review-memory/scripts/setup_memory.sh <프로젝트경로>
```

리뷰 메모리는 대상 프로젝트가 아니라 **mrt 저장소 안**(`<mrt>/.review-memory/<프로젝트명>/`)에 모인다.
스크립트 출력의 **마지막 줄이 이 프로젝트의 메모리 폴더 절대경로**다 — 아래 파일들은 그 경로 기준으로 읽는다
(literal `.review-memory/`가 대상 프로젝트에 있다고 가정하지 않는다). 이후 `<MEM>`은 그 경로를 가리킨다.

그 다음 반드시 읽는다:
- `<MEM>/context.md` — 사용자가 미리 등록한 배경/중점 사항/팀 규칙. 리뷰 우선순위에 직접 반영한다.
- `<MEM>/conventions.md` — 이 프로젝트의 반복 패턴/컨벤션. 리뷰 관점에 반영한다.
- `<MEM>/reviews/` 의 최근 2~3개 파일 — 최근 리뷰 맥락. 같은 작성자/같은 영역이면 특히 유용하다.

메모리가 비어 있으면(첫 리뷰) 그냥 진행하면 된다. 사용자가 리뷰 요청과 함께 배경 정보를 줬다면
그것도 이번 리뷰에 반영하고, "앞으로/항상" 같은 지속 의도가 보이면 context.md에 등록한다
(등록 방법은 review-memory 스킬 참조).

> 파일 읽기/쓰기는 셸(`cat`/`echo`) 대신 Read/Write 도구를 사용한다.

### 리뷰 필수 체크리스트 (하드 게이트)

`<MEM>/context.md`의 **"## 리뷰 필수 체크리스트"** 항목은 이 프로젝트에서 *놓치면 안 되는* 것들이다.
일반 참고가 아니라 **매 리뷰마다 PR과 대조해 통과 검사**하고, 결과 요약에 통과 여부를 명시한다
(걸리면 발견 사항으로, 이상 없으면 "체크리스트 N개 확인" 한 줄로). 변경 파일 경로에 해당하는
항목만 적용한다(모노레포면 예: `packages/jds/**`만 바뀌었으면 디자인시스템 항목).

- **PR 설명이 "후속 PR에서 처리"라고 미뤄도, 이 PR이 그 항목을 유발하면 면제되지 않는다.**
  예: 공개 API를 바꾸는 PR이 "changeset은 다음에"라고 해도, 변경 자체가 이 PR에 있으면 여기서 요구/확인한다.
  작성자의 계획을 그대로 받아들이지 말고, **규칙과 대조한 뒤** 동의/이견을 낸다. (실제로 이 한 줄을 안 지켜
  changeset 누락을 놓친 적이 있다 — 설명에 "후속에서 일괄"이라 적혀 있어 그대로 통과시킨 케이스.)

**첫 리뷰 감지는 `setup_memory.sh` 출력으로 한다.** 출력 첫 줄에 `FIRST_REVIEW=1`(이 프로젝트의 중앙
메모리에 아직 리뷰 기록이 없음 = 첫 리뷰) 또는 `FIRST_REVIEW=0`을 찍고, 마지막 줄에 메모리 경로를 찍는다.

`FIRST_REVIEW=1`이면 리뷰를 시작하기 전에 `AskUserQuestion`으로 이 프로젝트에서 *특히 중요한 게 무엇인지*
묻는다. 유형별로 핵심이 다르다 — 디자인시스템/라이브러리는 보통 **공개 API·버저닝(changeset)·a11y**,
서비스 앱은 **기존 동작 회귀·런타임 에러·로딩/에러 상태·데이터 정합성**이 핵심이다. 받은 답을 `<MEM>/context.md`의
"## 리뷰 필수 체크리스트"에 기록한 뒤 진행한다. 모노레포면 경로별(예: `packages/*` vs `apps/*`)로 나눠 묻고 기록한다.

## 2단계: PR 정보 수집

```bash
gh pr view <번호|URL> --json title,body,author,baseRefName,headRefName,headRefOid,files
gh pr diff <번호|URL>
```

`headRefOid`(PR 최신 커밋 SHA)는 결과 출력의 바로가기 링크에 쓰인다:
`https://github.com/<owner>/<repo>/blob/<headRefOid>/<파일경로>#L<시작>-L<끝>`

- PR이 지정되지 않았으면 `gh pr list`로 열린 PR을 보여주고 어떤 것인지 확인한다. 현재 브랜치에 연결된 PR이 있으면 그것을 기본 후보로 제시한다.
- diff만으로 판단이 어려운 부분은 저장소에서 관련 파일을 직접 읽는다. 변경된 함수의 호출처, 수정된 타입의 사용처 등 **diff 바깥의 영향**을 확인하는 것이 좋은 리뷰와 표면적인 리뷰의 차이다.

**기존 코멘트도 반드시 수집한다** — 다른 리뷰어가 이미 무엇을 지적했는지 모르고 쓴 리뷰는
중복 지적으로 신뢰를 잃는다:

```bash
# 대화 코멘트 + 리뷰 상태(승인/변경요청). body는 1500자로 잘라 토큰 낭비를 막는다
gh pr view <번호|URL> --json comments,reviews \
  --jq '{comments: [.comments[] | {author: .author.login, body: .body[0:1500]}], reviews: [.reviews[] | {author: .author.login, state, body: .body[0:1500]}]}'

# 인라인 리뷰 스레드 — resolved(해결됨) 여부까지 포함한다.
# REST(pulls/comments)에는 isResolved가 없으므로 GraphQL로 스레드 단위로 가져온다.
# isResolved=true / isOutdated=true 스레드는 이미 닫혔거나 라인이 사라진 것이다.
gh api graphql -f owner=<owner> -f repo=<repo> -F pr=<번호> -f query='
query($owner:String!,$repo:String!,$pr:Int!){
  repository(owner:$owner,name:$repo){
    pullRequest(number:$pr){
      reviewThreads(first:100){
        nodes{
          isResolved isOutdated
          comments(first:20){ nodes{ databaseId path line author{login} body } }
        }
      }
    }
  }
}' --jq '[.data.repository.pullRequest.reviewThreads.nodes[] | {
  isResolved, isOutdated,
  path: .comments.nodes[0].path, line: .comments.nodes[0].line,
  comments: [.comments.nodes[] | {id: .databaseId, author: .author.login, body: .body[0:1500]}]
}]'
```

> 스레드가 100개를 넘으면 `reviewThreads(first:100)`에 `pageInfo{ hasNextPage endCursor }`를 추가해 `after:` 커서로 페이지를 이어 받는다.

봇 코멘트(`vercel`, `coderabbitai[bot]`, `github-actions` 등)는 사람 코멘트와 구분한다 —
봇 지적 중 타당한 것은 참고하되, 사람 리뷰어의 코멘트가 리뷰 맥락의 중심이다.
봇의 장문 코멘트는 잘려도 무방하지만, **사람** 코멘트가 잘렸으면 그 코멘트만 따로 전체를 읽는다.

## 3단계: 리뷰

다음 관점으로 본다. context.md의 중점 사항과 메모리에서 읽은 컨벤션 위반은 가장 먼저 체크한다.

1. **정확성** — 버그, 엣지 케이스, 잘못된 로직, 동시성/비동기 문제
2. **프로젝트 컨벤션** — conventions.md의 패턴 + 코드베이스의 기존 스타일과의 일관성
3. **설계** — 변경이 기존 구조와 어울리는지, 더 단순한 방법이 있는지
4. **테스트** — 변경에 대한 테스트가 있는지, 깨지기 쉬운 부분이 커버되는지
5. **보안** — 입력 검증, 인증/인가, 민감 정보 노출

심각도가 낮은 스타일 지적을 나열하는 것보다, 확신 있는 소수의 발견이 더 가치 있다.
불확실한 것은 단정하지 말고 질문으로 분류한다.

**기존 코멘트와의 교차 검토** — 2단계에서 수집한 코멘트를 발견 사항과 대조한다:

- 다른 리뷰어가 이미 지적한 것과 같은 발견은 새 발견처럼 쓰지 않는다. 동의하면 해당
  스레드에 근거를 보강하는 **답글**을 제안하고, 다르게 보면 반박 근거와 함께 답글로 분류한다.
- **`isResolved: true`인 스레드는 이미 닫힌 것이므로 재지적하지 않는다** — 답글이나 코드 수정이
  없어도 마찬가지다. resolve된 지적을 다시 새 발견처럼 쓰면 신뢰를 잃는다. 다만 resolve됐지만
  실제 코드에 문제가 그대로 남아 있다고 확신하면, 새 발견이 아니라 **해당 스레드에 "여전히 남아
  있는 것 같다"는 답글**로 분류하고 근거를 단다.
- `isOutdated: true`(코멘트가 달린 라인이 이후 변경으로 사라짐)는 후속 커밋에서 손댄 신호다 —
  최신 diff를 확인하고, 문제가 해소됐으면 재지적하지 않는다.
- 기존 코멘트에 작성자가 이미 답했거나 후속 커밋으로 수정했으면 재지적하지 않는다
  (코멘트의 라인이 최신 diff에서 어떻게 바뀌었는지 확인).
- **아직 열려 있는(`isResolved: false`) 기존 지적** 중 이번 변경에 실제로 중요해 보이는 것이
  있으면 결과에서 짚어준다.

## 4단계: 결과 출력

리뷰를 읽는 사람이 (1) 문제가 된 **실제 코드**를 바로 보고, (2) **클릭 한 번**으로 GitHub의 해당 위치로 이동하고, (3) 동의하면 **코멘트를 복사해서 바로 달 수 있어야** 한다. 그래서 발견 사항은 한 줄 나열이 아니라 항목별 블록으로 쓴다. ALWAYS 이 구조를 사용한다:

````markdown
# PR #<번호> 리뷰: <제목>

## 요약
(이 PR이 무엇을 하는지 + 전체적인 평가 2~3문장)

## 발견 사항

### 🔴 반드시 수정 (머지 전 해결 필요)

#### 1. <문제를 한 줄로 요약한 제목>
📍 [`파일경로:시작-끝`](https://github.com/<owner>/<repo>/blob/<headRefOid>/<파일경로>#L<시작>-L<끝>)

```<언어>
<해당 코드 영역 — 문제가 보이는 최소한의 줄들>
```

**문제**: 왜 문제인지 1~3문장.
**제안**: 어떻게 고칠지. 가능하면 수정 코드를 보여준다.

**추천 코멘트** (복사해서 GitHub에 붙여넣기):
> 리뷰어가 실제로 달 만한 톤의 문구. 한 줄 수정으로 끝나는 경우
> ```suggestion 블록을 포함하면 작성자가 원클릭으로 반영할 수 있다.

### 🟡 권장 (고치면 좋음)
(같은 블록 구조, 추천 코멘트 포함)

### 💬 질문 / 확인 필요
(같은 블록 구조 — 추천 코멘트는 질문 형태로)

### 🔁 기존 코멘트 관련 (있는 경우만)

#### 1. <리뷰어>의 지적: <한 줄 요약>
📍 해당 스레드 위치 + 코멘트 링크

**상황**: 누가 무엇을 지적했고 현재 어떤 상태인지 1~2문장.
**내 판단**: 동의/이견과 그 근거.

**추천 답글** (해당 스레드에):
> 동의면 근거를 보강하는 답글, 이견이면 반박 근거를 담은 답글.

## 잘된 점
(있다면 1~2개. 억지로 만들지 않는다)
````

- 코드 블록은 diff 전체가 아니라 **문제가 보이는 최소 범위**만. 맥락이 필요하면 주변 2~3줄까지.
- 바로가기 링크는 반드시 `headRefOid` 고정 SHA를 쓴다 — 브랜치명 링크는 푸시되면 라인이 어긋난다.
- 추천 코멘트는 그대로 복사해 쓸 수 있는 완성문으로 쓴다. 기계적인 지적 톤이 아니라 동료에게 말하는 톤으로.

### 게시 코멘트 본문의 목소리 (페르소나)

위 채팅 요약의 🔴/🟡 블록·표는 **사용자가 빠르게 분류하라고** 있는 구조다. 하지만 **GitHub에 실제로 박히는 코멘트 본문**(추천 코멘트 / 5단계 게시 본문)은 받는 동료가 읽는 글이므로, 봇처럼 보이지 않게 사람이 직접 쓴 톤으로 적는다. 채팅 요약 톤과 게시 본문 톤을 **같게 쓰지 말 것** — 요약은 구조적으로, 게시 본문은 아래 목소리로.

이 저장소 리뷰어(Zero-1016)의 실제 말투에서 뽑은 기준:

- **담백한 존댓말.** 자주 쓰는 어미: `~있어 보여요`, `~좋아보여요`, `~있어서요`, `~싶습니다`, `~궁금해요`, `~네요`. 물결(`~`)은 가끔 한 번이면 충분.
- **단정보다 제안·질문으로.** "의도하신 건지 궁금해요", "굳이 분리할 필요가 있을까 싶습니다"처럼 여지를 남긴다. 확신이 약하면 질문으로.
- **거의 항상 코드로 보여준다.** 한 줄~몇 줄 수정이면 ` ```suggestion `(원클릭 반영), 그 외엔 ` ```ts `/` ```diff `, 시각 차이는 as-is/to-be 표나 이미지, 강한 경고만 `> [!CAUTION]`.
- **이모지 기본 0개.** 정말 자연스러운 자리에 어쩌다 하나 정도. 절대 뿌리지 않는다.
- **칭찬·인사치레 금지.** "고생하셨어요 / 좋은 PR이에요" 같은 말은 게시 본문에 넣지 않는다. (감사 표현은 내가 답변자 입장일 때만.)
- **골격 라벨 금지.** 본문에 `**문제:**`, `**제안:**`, `🔴 제목` 같은 템플릿 머리말을 쓰지 않는다. `<details>` 접기도 봇 느낌이라 게시 본문엔 쓰지 않는다.
- 구조는 **요지 한 줄 → 코드 → 짧은 근거/맥락** 흐름으로 자연스럽게 푼다.
- 과거 사례는 PR 번호를 기계적으로 박지 말고("#469처럼"), 맥락으로 녹이거나("예전에 Toast에서 같은 방식으로 색이 틀어졌던 적이 있어서요") 관련 commit/문서 링크로 단다.

GitHub에 코멘트를 **게시**하는 것은 사용자가 명시적으로 요청할 때만 한다(5단계). 기본 동작은 위 결과(추천 코멘트 포함)를 대화에 보여주는 것이다.

## 5단계: 코멘트 게시 (사용자가 요청할 때만)

사용자가 "코멘트 달아줘", "2번만 달아줘" 처럼 게시를 요청하면:

### 5-0. 게시 방식 선택 (AskUserQuestion)

게시 방식은 결과가 크게 다르므로(특히 작성자에게 즉시 노출되는지 여부) **반드시 사용자가 고르게 한다.**
사용자가 요청 문구에서 방식을 이미 지정했으면(예: "초안으로 달아둬", "pending으로", "바로 인라인으로
달아줘", "일반 코멘트로") 질문을 건너뛰고 그 방식으로 진행한다. 지정이 없으면 `AskUserQuestion`으로 묻는다:

- **헤더**: `게시 방식`
- **옵션**:
  1. `초안(Pending) 리뷰 (추천)` — 인라인 코멘트들을 하나의 리뷰로 모아 **PENDING** 상태로 만든다.
     제출(Submit) 전까지는 **리뷰어 본인에게만 보이고 작성자에게는 안 보인다.** 검토 후 GitHub UI나
     명령으로 제출/폐기할 수 있다. (보내기 전에 모아서 다듬고 싶을 때)
  2. `인라인 코멘트 즉시 게시` — 각 발견을 해당 라인에 **바로 공개**로 단다. 작성자에게 즉시 알림이 간다.
  3. `일반 코멘트 한 건으로 게시` — PR 전체에 단일 코멘트로 **바로 공개** 게시한다.

> 답변에 "항상/앞으로/기본으로" 같은 지속 의도가 보이면(예: "앞으로 늘 초안으로") 그 선호를
> `<MEM>/context.md`의 "팀 규칙"에 등록해, 다음 리뷰부터는 질문 없이 그 방식을 기본으로 쓴다.
> context.md에 이미 기본 방식이 등록돼 있으면 5-0 질문을 건너뛰고 그 방식으로 진행한다.

### 5-1. 게시 전 확인 및 본문 작성

1. **게시 전 확인** — 게시할 코멘트의 최종 본문과 위치를 보여주고 사용자 확인을 받는다.
   (사용자가 "전부 달아줘"라고 했고 본문이 이미 4단계 결과에 그대로 있다면 생략 가능)
2. 본문은 따옴표/개행 깨짐을 피하기 위해 임시 파일(또는 JSON 페이로드)로 작성해서 전달한다.

### 5-2. 방식별 명령

```bash
# [방식 1] 초안(Pending) 리뷰 — event를 생략하면 PENDING으로 생성된다(제출 전 비공개).
#   인라인 코멘트들을 comments[] 배열로 한 번에 담는다. body는 리뷰 총평.
#   본문에 한글/백틱/개행이 많으면 JSON 페이로드 파일을 만들어 --input으로 넘기는 것이 안전하다.
#   payload 예: {"commit_id":"<headRefOid>","body":"<총평>",
#     "comments":[{"path":"<파일>","line":<끝라인>,"side":"RIGHT","body":"<본문>"}, ...]}
gh api repos/<owner>/<repo>/pulls/<번호>/reviews --input /tmp/review_payload.json \
  --jq '{id, state, html_url}'
#   → state가 PENDING인지 확인하고 html_url을 사용자에게 보여준다.
#   제출:  gh api repos/<owner>/<repo>/pulls/<번호>/reviews/<review_id>/events -f event=COMMENT
#   폐기:  gh api -X DELETE repos/<owner>/<repo>/pulls/<번호>/reviews/<review_id>

# [방식 2] 인라인 코멘트 즉시 게시 (commit_id는 2단계의 headRefOid)
gh api repos/<owner>/<repo>/pulls/<번호>/comments \
  -F body=@/tmp/comment.md -f commit_id=<headRefOid> \
  -f path='<파일경로>' -F line=<끝라인> -f side=RIGHT
# 여러 줄 범위면 -F start_line=<시작라인> -f start_side=RIGHT 추가

# [방식 3] PR 전체에 대한 일반 코멘트 즉시 게시
gh pr comment <번호|URL> --body-file /tmp/comment.md

# 기존 스레드에 답글 (comment_id는 2단계에서 수집한 인라인 코멘트의 id)
gh api repos/<owner>/<repo>/pulls/<번호>/comments/<comment_id>/replies \
  -F body=@/tmp/comment.md
```

- 인라인 코멘트(방식 1·2)의 `line`은 **diff에 포함된 라인**이어야 한다(변경되지 않은 라인에는 달 수 없음). 실패하면 가장 가까운 변경 라인으로 옮기거나 일반 코멘트로 대체한다. 방식 1은 `comments[]` 중 한 줄이라도 라인이 틀리면 전체 호출이 실패하므로 라인 번호를 특히 주의한다.
- 게시 후 생성된 리뷰/코멘트 URL을 사용자에게 보여준다. 방식 1은 **아직 제출되지 않았음(PENDING)** 을 명확히 알린다.
- 승인/변경요청 같은 **리뷰 판정**(`gh pr review --approve/--request-changes`, 또는 reviews API의 `event=APPROVE/REQUEST_CHANGES`)은 사용자가 그 단어로 직접 요청할 때만 한다.

## 6단계: 메모리 저장

리뷰를 보여준 뒤, 결과를 기록한다:

1. `<MEM>/reviews/<YYYY-MM-DD>-pr-<번호>.md` 에 저장 — 위 리뷰 결과에 PR 제목/작성자/날짜를 헤더로 붙인 것. 다음 리뷰 때 맥락으로 쓸 수 있게 간결하게.
2. **conventions.md 업데이트 — 반드시 사용자 확인 후에만 기록한다.** 자동으로 쓰지 않는다.
   이번 발견 중 "이 프로젝트에서 또 나올 법한 것"이 있는지 보고, **후보가 있을 때만** 사용자에게
   제안한 뒤 승인받은 항목만 conventions.md에 반영한다:
   - 후보를 1~2줄로 보여준다 (새 항목이면 "무엇/왜/어떻게 피하는지", 기존 항목이면 "어디에 횟수/날짜 추가").
   - `AskUserQuestion`(또는 간단한 승인 요청)으로 **추가할지 / 빼고 갈지 / 문구를 다듬을지**를 묻는다.
   - 승인된 것만 쓴다. 사용자가 빼라면 안 쓴다.
   - **후보 기준(이게 있어야 후보로 올린다)**: 과거에도 나온 반복 이슈, 또는 프로젝트 고유의 새 패턴.
     일회성 실수·일반론·한 번뿐인 스타일 지적은 후보로 올리지 않는다 — conventions.md가 잡동사니가 되면 읽는 의미가 없다.
   - 후보가 없으면 아무것도 묻지 않고 넘어간다. (1번의 reviews/ 파일은 원본 로그이므로 확인 없이 저장한다.)

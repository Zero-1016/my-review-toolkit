# 브랜치 리뷰 (base 대비, 심각도 분류)

현재 브랜치에서 한 작업 전체를 base 브랜치 기준으로 리뷰한다. 커밋 단위가 아니라
**브랜치에서 달라진 모든 것**(커밋된 것 + 미커밋 작업)이 대상이며, 발견 사항은
반드시 High/Medium/Low 세 단계 심각도로 분류한다.
pr-review/local-code-review와 같은 프로젝트 메모리를 공유한다.

## 1단계: 메모리 준비 및 로드

```bash
bash ~/.claude/skills/review-memory/scripts/setup_memory.sh <프로젝트경로>
```

리뷰 메모리는 대상 프로젝트가 아니라 **mrt 저장소 안**(`<mrt>/.review-memory/<프로젝트명>/`)에 모인다.
스크립트 출력의 **마지막 줄이 이 프로젝트의 메모리 폴더 절대경로**(`<MEM>`)다 — 아래 파일들은 그 경로 기준으로 읽는다.

그 다음 반드시 읽는다:
- `<MEM>/conventions.md` — 반복 패턴/컨벤션
- `<MEM>/reviews/` 의 최근 2~3개 — 최근 리뷰 맥락

> 파일 읽기/쓰기는 셸(`cat`/`echo`) 대신 Read/Write 도구를 사용한다.

## 2단계: base 브랜치 결정 및 diff 수집

```bash
# base 브랜치: 사용자가 지정하면 그것, 아니면 origin/HEAD → main → master 순으로 탐지
git symbolic-ref refs/remotes/origin/HEAD --short 2>/dev/null \
  || git branch -l main master

# 분기점(merge-base) 기준으로 브랜치의 모든 변경 (커밋 + 미커밋 포함)
git merge-base <base> HEAD          # 분기점 SHA
git diff --stat <분기점SHA>          # 변경 규모 파악
git diff <분기점SHA>                 # 전체 diff (작업 트리 기준)
git log --oneline <base>..HEAD      # 브랜치 커밋 목록
```

- merge-base를 쓰는 이유: base가 앞서 나갔어도 **내가 바꾼 것만** 보기 위해서다. `git diff <base>`를 직접 쓰면 base의 새 커밋이 역방향 변경으로 섞인다.
- 미커밋 변경도 작업의 일부이므로 기본 포함한다. 사용자가 "커밋된 것만"이라고 하면 `<분기점SHA>..HEAD`로 제한한다.
- 현재 브랜치가 base 자체라면 그 사실을 알리고 미커밋 변경만 리뷰한다.

diff에 새 파일이 있으면 전체를 읽고, 수정된 파일은 변경 주변 맥락을 읽는다.
**diff 바깥의 영향**(수정·삭제된 함수의 다른 호출처, 시그니처 변경으로 깨질 사용처, base에 있던 동작의 회귀)을 확인하는 것이 좋은 리뷰와 표면적인 리뷰의 차이다.

## 3단계: 리뷰 및 심각도 분류

다음 관점으로 본다. 메모리에서 읽은 컨벤션 위반은 가장 먼저 체크한다:

1. **정확성** — 버그, 엣지 케이스, 잘못된 로직, 동시성/비동기 문제
2. **회귀** — base에 있던 동작/에러 처리/검증이 사라지지 않았는지
3. **프로젝트 컨벤션** — conventions.md 패턴 + 기존 코드 스타일과의 일관성
4. **보안** — 입력 검증, 인증/인가, 민감 정보 노출
5. **테스트** — 새 로직에 테스트가 있는지
6. **잔재물** — 디버그 코드, 주석 처리된 코드, 임시 하드코딩, 무관한 변경

모든 발견은 아래 기준으로 심각도를 부여한다. 애매하면 한 단계 낮춘다 —
High가 양치기 늑대가 되면 분류 자체가 무의미해진다:

| 심각도 | 기준 | 예시 |
|--------|------|------|
| 🔴 **High** | 잘못된 결과·데이터 손상·보안 취약점·기존 동작의 회귀. 머지 전 반드시 수정 | 계산 오류, 에러 처리 삭제, 인젝션, 무한 루프 |
| 🟡 **Medium** | 특정 조건에서만 터지는 잠복 버그, 성능 문제, 컨벤션 위반, 테스트 누락. 고치기를 권장 | off-by-one(가드로 가려진), 루프 내 순차 await, `==` 사용 |
| 🟢 **Low** | 동작에는 영향 없는 정리거리. 선택 사항 | console.log 잔재, 네이밍, 사소한 중복 |

확신 없는 것은 단정하지 말고 해당 심각도 항목 안에서 질문 형태로 쓴다.
심각도가 낮은 지적을 늘어놓기보다 확신 있는 소수의 발견이 가치 있다.

## 4단계: 결과 출력

읽는 사람이 (1) 문제가 된 **실제 코드**를 바로 보고, (2) `파일경로:라인`을 **클릭해서 에디터로 이동**할 수 있어야 한다. ALWAYS 이 구조를 사용한다:

````markdown
# 브랜치 리뷰: <브랜치명> (base: <base브랜치> 대비)

## 요약
(이 브랜치가 무엇을 하는지 + 전체 평가 2~3문장. High N건 / Medium N건 / Low N건)

## 발견 사항

### 🔴 High — 머지 전 반드시 수정

#### 1. <문제를 한 줄로 요약한 제목>
📍 `상대경로/파일.js:시작-끝`

```<언어>
<해당 코드 영역 — 문제가 보이는 최소한의 줄들>
```

**문제**: 왜 이 심각도인지가 드러나게 1~3문장.
**수정 제안**:
```<언어>
<그대로 적용 가능한 고친 코드>
```

### 🟡 Medium — 수정 권장
(같은 블록 구조)

### 🟢 Low — 선택
(같은 블록 구조, 수정 제안은 한 줄 설명으로 충분하면 코드 생략 가능)
````

- 해당 심각도에 발견이 없으면 그 섹션은 "없음" 한 줄로 남긴다 — 섹션 자체를 빼지 않는다 (안 본 것인지 없는 것인지 구분되도록).
- 위치는 `파일경로:라인` 형식을 지킨다 — 터미널/IDE에서 클릭하면 해당 위치로 이동한다.
- 코드 블록은 문제가 보이는 최소 범위만. 맥락이 필요하면 주변 2~3줄까지.

수정은 사용자가 요청할 때만 적용한다. 기본 동작은 발견 사항 보고다.

## 5단계: 메모리 저장

1. `<MEM>/reviews/<오늘날짜>-branch-<브랜치명>.md` 에 리뷰 결과 저장 (날짜/브랜치/base/심각도별 건수를 헤더로)
2. **conventions.md 업데이트 판단** — pr-review와 같은 기준:
   - 반복해서 나오는 이슈 → 기존 항목에 횟수/날짜 추가
   - 프로젝트 고유의 새 패턴 → 새 항목 추가
   - 일회성 실수는 기록하지 않는다

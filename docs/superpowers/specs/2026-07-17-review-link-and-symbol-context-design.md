# 리뷰 스킬 보강: 코드 위치 링크(하드 게이트) + 정규화/검증 일관성 역추적

- 날짜: 2026-07-17
- 대상 스킬: `pr-review`, `branch-review`, `local-code-review`
- 관련: `skills/*/SKILL.md`

## 배경

리뷰 스킬을 쓰다 두 가지 부족함이 반복됐다.

1. **코드 위치 링크 누락.** 리뷰를 읽는 사람이 지적된 위치로 바로 가야 하는데,
   `pr-review`는 permalink가 출력 템플릿에만 있고 "반드시 붙인다"는 강제가 없어
   실제 리뷰에서 링크가 빠진 케이스가 있었다. `branch-review`·`local-code-review`는
   웹 링크 규칙 자체가 없다(`파일:라인`만).
2. **diff만 봐서 심볼의 맥락이 부족.** diff에 등장한 값이 어딘가에서 정제/검증
   (trim·소문자화·null 가드·parse·sanitize·기본값)되면, 리뷰어는 "이 값은 이미 정제된
   값"이라고 가정하기 쉽다. 하지만 선언부나 형제 대입 경로에서 그 처리가 빠져 있을 수 있다.
   실제로 "한쪽 경로에선 trim된 값으로 비교하는데 선언부에선 trim이 누락"된 걸
   diff만 봐서 놓친 적이 있다.

## 목표

- **기능 1 — 위치 링크 하드 게이트:** 세 스킬 모두 *모든 발견 사항*에 위치 줄(`📍`)을
  필수로 강제한다. 링크 종류만 스킬/상태별로 결정한다.
- **기능 2 — 정규화/검증 일관성 역추적:** diff에 나온 값이 정제/검증되면, 그 심볼을
  diff 밖으로 거꾸로 추적해 선언부·형제 경로가 같은 처리를 하는지 확인하는 지침을 추가한다.

비목표(YAGNI): diff에 나온 *모든* 심볼을 추적하지 않는다(토큰/시간 비용). 정제/검증이
걸린 값에만 발동한다. 공유 include 메커니즘은 만들지 않는다 — 세 SKILL.md에 일관된
문구를 직접 넣는다(이 저장소는 이미 "diff 바깥 영향" 문단을 세 파일에 중복 보유하는 패턴).

## 기능 1: 위치 링크 하드 게이트

### 공통 규칙 (세 스킬 출력 단계에 삽입)

> **위치 줄(`📍`)은 모든 발견 사항에 필수다.** 링크가 없으면 그 발견은 미완성으로 본다.
> 링크 소스를 못 구하면(예: `headRefOid` 미확보, 원격 없음) 먼저 확보를 시도하고,
> 정말 불가능할 때만 `파일:라인`으로 대체한다.

### 스킬별 링크 종류

| 스킬 | 링크 종류 |
|------|-----------|
| `pr-review` | 기존 `headRefOid` 기반 blob permalink 유지. **모든 발견 블록에 필수**로 명시. `headRefOid` 미확보 시 링크 없이 진행하지 말고 먼저 확보(2단계에서 이미 수집). |
| `branch-review` | 라인이 **커밋+푸시**됐으면 웹 permalink, 아니면 `파일:라인`. |
| `local-code-review` | 동일 이원화. 미커밋 변경이 기본 대상이라 대부분 `파일:라인`이지만, 커밋+푸시된 라인이면 permalink. |

### branch/local의 웹 permalink 규칙

원격 permalink 베이스와 링크 대상 SHA를 리뷰 diff 수집 단계에서 함께 파악한다.

```bash
# 원격 base URL (없으면 웹 링크 불가 → 파일:라인)
git remote get-url origin
#   git@github.com:owner/repo.git  → https://github.com/owner/repo
#   https://github.com/owner/repo.git → https://github.com/owner/repo
#   (.git 접미사 제거, ssh 형식은 host:path → host/path 로 변환)

# 브랜치가 원격에 푸시돼 있는지 + 푸시된 팁 SHA
git rev-parse --abbrev-ref --symbolic-full-name @{push} 2>/dev/null   # 업스트림 존재 확인
git rev-parse @{push} 2>/dev/null                                     # 푸시된 팁 SHA
```

permalink 형식:

```
https://<host>/<owner>/<repo>/blob/<푸시된SHA>/<파일경로>#L<시작>-L<끝>
```

판정 규칙:

- **웹 permalink를 쓰는 조건:** 해당 라인이 커밋되어 있고 그 커밋이 원격에 푸시돼 있어,
  푸시된 SHA에서 그 파일·라인이 지금과 동일하다. 실무 판정: **작업트리가 깨끗하고
  `HEAD == @{push}`** 이면 모든 라인을 HEAD(=푸시된) SHA로 링크한다.
- **`파일:라인`으로 두는 경우:** 미커밋 변경, 미푸시 커밋, 원격 없음, 또는 혼재 상태로
  라인이 푸시된 SHA에 없을 수 있을 때. (웹에 그 라인이 없으면 링크가 깨진다.)
- **호스트:** GitHub/GitLab은 `#L시작-L끝` 앵커. Bitbucket 등 앵커 문법이 다르면
  안전하게 `파일:라인`으로 폴백한다.

SHA 고정이 핵심이다 — 브랜치명 링크는 이후 푸시로 라인이 어긋난다(`pr-review`가
`headRefOid`를 쓰는 것과 같은 이유).

## 기능 2: 정규화/검증 일관성 역추적

세 스킬의 리뷰 단계(리뷰 관점 목록 뒤)에 공통 문단으로 삽입한다.

### 삽입 문구 (세 스킬 공통)

> **정규화/검증 일관성 (선언부 역추적).** diff에 나온 값이 어딘가에서 정제/검증되면
> (trim·`toLowerCase`·null/undefined 가드·parse·sanitize·기본값 대입 등), 그 심볼을
> diff 밖으로 **거꾸로** 추적해 선언부와 형제 대입 경로가 같은 처리를 하는지 확인한다.
> diff만 보면 "이 값은 이미 정제된 값"이라고 가정하기 쉽지만, 실제 선언부/다른 경로에서
> 그 처리가 빠져 있을 수 있다.
>
> - **트리거:** 모든 심볼이 아니라 **정제/검증이 걸린 값**에만 — 비교, 조회 키, 저장,
>   외부 전송 직전에 정제된 심볼.
> - **방법:** `git grep`/파일 읽기로 변수·필드·함수 인자의 **선언부와 다른 대입 지점**을
>   찾고, 필요하면 `git log -L :<심볼>:<파일>` / `git blame`으로 그 처리가 언제·왜
>   들어갔는지(또는 빠졌는지) 본다.
> - **표현:** 확신이 없으면 단정하지 말고 질문으로 — "선언부에서 trim이 누락된 것 같은데
>   의도하신 건가요?"

이는 기존 "diff 바깥 영향(호출처 확인)" 문단과 짝을 이룬다: 기존 문단은 **정방향**
(변경 → 사용처), 이 문단은 **역방향**(변경된 값 → 선언부/정규화 지점).

## 검증

스킬은 코드가 아니라 지시문이므로 자동 테스트가 없다. 검증은:

1. 세 SKILL.md의 frontmatter(`name`/`description`)가 5-에이전트 공용 형식을 유지하는지.
2. 추가한 셸 recipe가 `bash`/`git`만 쓰는지(특정 에이전트 전용 도구 없음).
3. 문구가 세 파일에서 일관된지(기능 2 문단, 기능 1 게이트 문구).
4. `changelog` 훅이 커밋 시 `[Unreleased]`에 자동 기록.

## 위험 / 완화

- **깨진 웹 링크:** 미푸시 라인을 permalink로 걸면 404/라인 어긋남. → "커밋+푸시 확정"
  조건을 보수적으로 잡고 애매하면 `파일:라인`으로 폴백.
- **기능 2 과발동:** 모든 심볼을 추적하면 리뷰가 느려짐. → "정제/검증이 걸린 값에만"으로
  트리거를 좁힘.
- **호스트 다양성:** GitHub/GitLab 외 앵커 문법 차이. → 미지원 호스트는 `파일:라인` 폴백.

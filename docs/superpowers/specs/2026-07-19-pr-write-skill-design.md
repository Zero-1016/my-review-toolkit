# PR 작성 스킬(`pr-write`) + PR 템플릿 설계

- 날짜: 2026-07-19
- 대상: 새 스킬 `skills/pr-write/`, 새 파일 `.github/PULL_REQUEST_TEMPLATE.md`
- 관련: `install.sh`, `README.md`, `AGENTS.md`, `CHANGELOG.md`, `scripts/hooks/doc-freshness.sh`

## 배경

기존 스킬 5종은 전부 **리뷰** 쪽이다(PR/브랜치/로컬 리뷰 + 메모리 + 체인지로그). 정작 PR을
**작성**하는 단계는 스킬이 없어서 에이전트가 즉석에서 제목·본문을 짰고, 두 가지 문제가 반복됐다.

1. **제목/본문 골격이 매번 제각각** — 읽는 사람이 "무엇을·왜" 빠르게 파악하기 어렵다.
2. **GitHub 렌더에서 줄바꿈이 깨짐** — 본문 문단을 80자에서 하드랩(문단 중간 수동 개행)하면,
   GitHub는 PR/코멘트에서 **문단 내부의 단일 개행을 실제 줄바꿈으로 렌더**하기 때문에
   문장이 어정쩡하게 끊겨 보인다. (이 저장소 PR #1이 실제로 그렇게 나왔다.)

## 목표

- 커밋/diff를 읽어 **명확한 제목 + 정형 본문**을 만들고, 사용자 확인 후 `gh`로 PR을
  생성/수정까지 하는 스킬 `pr-write`를 추가한다.
- GitHub에 그대로 올려도 줄바꿈이 깨지지 않는 **하드 규칙**을 스킬이 강제한다.
- 사람이 손으로 PR을 열 때도 같은 골격을 쓰도록 `.github/PULL_REQUEST_TEMPLATE.md`를 둔다.

비목표(YAGNI): 리뷰(품질 점검)는 하지 않는다 — 그건 `local-code-review`/`branch-review`
몫이다. 커밋 자동 생성/스쿼시도 하지 않는다. 라벨·리뷰어 지정 자동화도 넣지 않는다.

## 스킬 골격 (`skills/pr-write/SKILL.md`)

frontmatter는 5-에이전트 공용 형식(한국어 `name`/`description`, 트리거 문구 포함),
본문은 `bash`/`git`/`gh`만 사용.

1. **변경 파악** — 현재 브랜치·base 브랜치 감지(`main`/`master`), `git log <base>..HEAD`,
   `git diff --stat`, 원격/푸시 여부(`@{push}`). 기존 PR이 있으면(`gh pr view`) 수정 모드.
2. **제목 규칙** — Conventional Commits식 `type(scope): 요약`. 한 줄, 72자 이내, 명령형,
   마침표 없음. 커밋이 하나면 그 요지를, 여러 개면 공통 주제를 요약.
3. **본문 템플릿** — 질문형 헤더 + 이모지: `## 📌 어떤 것이 변경되었나요? / ## 💡 왜 이
   변경이 필요한가요? / ## 📝 변경 내역 / ## ✅ 검증 / ## 🔗 참고(선택)`. 해당 없는 섹션은
   통째로 생략한다(빈 섹션 금지). "왜"에는 해결하는 문제/배경을 명확히.
4. **줄바꿈 하드 규칙(핵심)** — 문단 텍스트는 **하드랩 금지, 한 문단 = 한 줄**.
   문단 구분은 빈 줄. bullet도 한 항목 한 줄. 코드·로그·명령만 ` ``` ` 블록. 이유(GitHub가
   문단 내 단일 개행을 줄바꿈으로 렌더)를 명시해 에이전트가 규칙을 이해하고 지키게 한다.
5. **생성/수정** — 본문은 셸 히어독이 아니라 **파일로 써서** `--body-file`로 넘긴다
   (히어독 하드랩이 문제의 원인이었으므로). 신규는 `gh pr create --base <base> --body-file`,
   기존 PR은 `gh pr edit <n> --title … --body-file`. 실행 전 제목·본문을 사용자에게 보이고
   확인받는다. 에이전트별 관례(세션 링크 트레일러 등)가 있으면 본문 끝에 붙인다.

## PR 템플릿 (`.github/PULL_REQUEST_TEMPLATE.md`)

스킬 본문의 4개 섹션과 동일한 골격을 담고, 상단에 "한 문단은 한 줄로 — 하드랩 금지"
주석 힌트를 둔다. 스킬은 이 파일이 있으면 그대로 채우는 것을 우선한다.

## 부수 변경

- `install.sh` — `SKILLS=(…)` 배열에 `pr-write` 추가(알파벳 순).
- `README.md` 스킬 목록 표에 `pr-write` 행 추가.
- `AGENTS.md` — 구조의 정본 스킬 목록에 `pr-write` 추가 + 첫 "스킬 N종" 숫자를 6으로.
  (doc-freshness 훅이 README·AGENTS 양쪽 언급과 "스킬 N종" 숫자를 검사한다.)
- `CHANGELOG.md` `[Unreleased]`에 쉬운 문장으로 기록.

## 검증

스킬은 지시문이라 자동 테스트가 없다. 검증은:

1. `bash scripts/hooks/doc-freshness.sh --report`가 통과(드리프트 없음).
2. `pr-write`로 만든 실제 PR(이 저장소 PR #1 리파인)이 GitHub에서 줄바꿈 깨짐 없이 렌더.
3. frontmatter가 5-에이전트 공용 형식 유지, 본문이 `bash`/`git`/`gh`만 사용.

## 위험 / 완화

- **줄바꿈 규칙을 어겨 다시 하드랩:** → 본문을 파일로 쓰고 규칙을 스킬 상단에 굵게 명시.
- **base 브랜치 오판:** → `gh repo view`/기본 브랜치 조회로 확인, 애매하면 사용자에게 확인.
- **리뷰 스킬과 혼동:** → description에 "작성 전용, 품질 점검은 리뷰 스킬" 경계를 명시.

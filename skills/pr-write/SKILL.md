# PR 작성

현재 브랜치의 변경을 사람이 **한눈에 "무엇을·왜"** 파악할 수 있는 PR로 만든다. 핵심은
두 가지: (1) 제목·본문을 매번 같은 골격으로 통일하고, (2) GitHub에 올려도 **줄바꿈이
깨지지 않게** 쓰는 것. 이 스킬은 작성 전용이다 — 코드 품질을 점검하려면 리뷰 스킬을 쓴다.

## 0단계: 하드 규칙 — 줄바꿈 (제일 먼저 읽는다)

GitHub는 PR 본문·코멘트에서 **문단 내부의 단일 개행을 실제 줄바꿈(`<br>`)으로 렌더**한다.
그래서 본문 문단을 80자에서 하드랩(문단 중간에 수동 개행)하면 문장이 어정쩡하게 끊겨 보인다.
따라서:

- **문단 텍스트는 하드랩하지 않는다. 한 문단 = 한 줄.** 길어도 한 줄로 쭉 쓴다(에디터에서
  줄이 넘쳐 보여도 GitHub가 알아서 감싼다).
- **문단 구분은 빈 줄** 하나로 한다.
- **bullet도 한 항목 = 한 줄.** 항목 안에서 개행하지 않는다.
- 여러 줄이 필요한 건 **코드·로그·명령뿐** — 그건 ` ``` ` 코드블록 안에 둔다.
- 본문을 만들 때 셸 히어독으로 하드랩해 만들지 말고, **파일에 쓴 뒤 `--body-file`로 넘긴다**
  (히어독 하드랩이 바로 이 깨짐의 원인이다).

## 1단계: 변경 파악

base 브랜치와 이 브랜치의 커밋/변경을 확인한다.

```bash
CUR="$(git rev-parse --abbrev-ref HEAD)"

# base 브랜치 감지: 원격 기본 브랜치 → 없으면 main/master 순.
BASE="$(gh repo view --json defaultBranchRef -q .defaultBranchRef.name 2>/dev/null)"
[ -n "$BASE" ] || { git show-ref -q --verify refs/heads/main && BASE=main || BASE=master; }

git log "$BASE..$CUR" --oneline          # 이 브랜치가 쌓은 커밋
git diff "$BASE...$CUR" --stat           # 무엇이 바뀌었나 (3-dot: merge-base 기준)
git rev-parse --abbrev-ref --symbolic-full-name @{push} 2>/dev/null || echo "(미푸시)"

# 이미 이 브랜치로 열린 PR이 있으면 → 신규 생성이 아니라 '수정' 모드.
gh pr view --json number,title,url 2>/dev/null || echo "(열린 PR 없음)"
```

- 커밋과 diff를 **실제로 읽어** 무엇이·왜 바뀌었는지 파악한다. 커밋 메시지만 믿지 말고
  diff로 확인한다.
- 열린 PR이 있으면 4단계에서 `create`가 아니라 `edit`로 간다.
- 브랜치가 아직 원격에 없으면 생성 전에 push가 필요하다(4단계).

## 2단계: 제목 만들기

**Conventional Commits 골격**으로 한 줄 제목을 만든다.

```
type(scope): 요약
```

- `type`: `feat` · `fix` · `docs` · `refactor` · `chore` · `test` · `perf` · `build` · `ci`.
- `scope`(선택): 바뀐 영역(패키지·모듈·파일군). 애매하면 생략.
- 규칙: **한 줄, 72자 이내, 명령형/현재형, 끝에 마침표 없음.** 커밋이 하나면 그 요지를,
  여러 개면 **공통 주제**를 요약한다(커밋 제목을 그대로 잇지 않는다).
- 이 저장소 커밋/브랜치 관례를 따른다. 한국어 요약이 관례면 한국어로 쓴다.

## 3단계: 본문 만들기 (템플릿)

저장소에 `.github/PULL_REQUEST_TEMPLATE.md`가 있으면 **그 골격을 그대로 채운다**. 없으면
아래 골격을 쓴다. **해당 없는 섹션은 통째로 생략**한다(빈 섹션을 남기지 않는다).

```markdown
## 무엇
<이 PR이 하는 일 한 줄 요약>

## 왜
<해결하는 문제나 배경. 리뷰어가 "왜 이걸 하는지" 납득되게. 이슈/이전 논의가 있으면 링크.>

## 변경 내역
- <커밋/영역 단위로 한 항목 한 줄>
- <...>

## 검증
<테스트·수동 확인 방법. 자동 테스트가 없으면 무엇을 어떻게 확인했는지.>

## 참고
<관련 이슈·후속 작업·스크린샷 등. 없으면 이 섹션 생략.>
```

- **"왜"가 가장 중요하다** — 사용자가 배경/문제를 알려줬거나 설계 문서(`docs/…/specs/`)가
  있으면 거기서 문제 정의를 끌어와 명확히 쓴다.
- UI 변경이면 "참고"에 스크린샷 자리를 만든다(`![before](...)` / `![after](...)`).
- **0단계 줄바꿈 규칙을 지킨다** — 각 문단은 한 줄, bullet은 한 항목 한 줄.

## 4단계: 확인 후 생성/수정

본문을 **파일로 쓴다**(히어독 하드랩 금지). 그 다음 제목·본문을 사용자에게 보여주고
확인을 받은 뒤에만 `gh`를 실행한다.

```bash
# 본문을 파일로 (에이전트의 파일 쓰기 도구로 작성하는 걸 권장 — 셸 히어독은 하드랩 위험).
BODY=<본문 파일 경로>

# (미푸시면 먼저) 브랜치 push
git push -u origin "$CUR"

# 신규 PR
gh pr create --base "$BASE" --head "$CUR" --title "<제목>" --body-file "$BODY"

# 기존 PR 수정 (1단계에서 열린 PR을 찾았을 때)
gh pr edit <번호> --title "<제목>" --body-file "$BODY"
```

- **실행 전 반드시 제목·본문 초안을 사용자에게 보이고 승인받는다.** PR 생성/수정은
  외부에 공개되는 동작이다.
- 에이전트/저장소에 본문 끝 관례(예: 세션 링크 트레일러)가 있으면 본문 파일 끝에 붙인다.
- 생성/수정 후 PR URL을 사용자에게 알려준다.

## 경계 (이 스킬이 하지 않는 것)

- **코드 품질 점검을 하지 않는다.** 버그·컨벤션 리뷰는 `local-code-review`(미커밋)·
  `branch-review`(브랜치 전체)·`pr-review`(GitHub PR) 몫이다.
- 커밋을 새로 만들거나 스쿼시하지 않는다. 라벨·리뷰어 지정도 사용자가 요청할 때만.

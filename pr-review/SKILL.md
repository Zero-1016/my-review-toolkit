# PR 리뷰

GitHub PR을 리뷰한다. 핵심 차별점은 **프로젝트 메모리**다: 리뷰 전에 이 프로젝트에서
과거에 발견된 패턴을 읽고, 리뷰 후에 새로 알게 된 것을 기록한다. 리뷰를 거듭할수록
이 프로젝트에 특화된 리뷰어가 되는 것이 목표다.

## 1단계: 메모리 준비 및 로드

```bash
RTK_SKIP=1 bash ~/.claude/skills/review-memory/scripts/setup_memory.sh <프로젝트경로>
```

그 다음 반드시 읽는다:
- `.review-memory/context.md` — 사용자가 미리 등록한 배경/중점 사항/팀 규칙. 리뷰 우선순위에 직접 반영한다.
- `.review-memory/conventions.md` — 이 프로젝트의 반복 패턴/컨벤션. 리뷰 관점에 반영한다.
- `.review-memory/reviews/` 의 최근 2~3개 파일 — 최근 리뷰 맥락. 같은 작성자/같은 영역이면 특히 유용하다.

메모리가 비어 있으면(첫 리뷰) 그냥 진행하면 된다. 사용자가 리뷰 요청과 함께 배경 정보를 줬다면
그것도 이번 리뷰에 반영하고, "앞으로/항상" 같은 지속 의도가 보이면 context.md에 등록한다
(등록 방법은 review-memory 스킬 참조).

> 환경 주의: 이 머신에서는 rtk 훅이 셸 명령을 재작성한다. 이 스킬의 모든 셸 명령
> (git/gh/스크립트)에는 `RTK_SKIP=1` 접두어를 붙여 재작성을 우회한다 — 훅 래퍼가
> 이 마커를 인식해 원본 명령을 그대로 실행한다. 그래도 `_safe_eval` 오류가 나면
> 해당 명령을 절대경로로 재시도하고(예: `/usr/bin/git`, `/opt/homebrew/bin/gh`, `/bin/ls`),
> 파일 읽기/쓰기는 Read/Write 도구를 사용한다.

## 2단계: PR 정보 수집

```bash
RTK_SKIP=1 gh pr view <번호|URL> --json title,body,author,baseRefName,headRefName,headRefOid,files
RTK_SKIP=1 gh pr diff <번호|URL>
```

`headRefOid`(PR 최신 커밋 SHA)는 결과 출력의 바로가기 링크에 쓰인다:
`https://github.com/<owner>/<repo>/blob/<headRefOid>/<파일경로>#L<시작>-L<끝>`

- PR이 지정되지 않았으면 `gh pr list`로 열린 PR을 보여주고 어떤 것인지 확인한다. 현재 브랜치에 연결된 PR이 있으면 그것을 기본 후보로 제시한다.
- diff만으로 판단이 어려운 부분은 저장소에서 관련 파일을 직접 읽는다. 변경된 함수의 호출처, 수정된 타입의 사용처 등 **diff 바깥의 영향**을 확인하는 것이 좋은 리뷰와 표면적인 리뷰의 차이다.

**기존 코멘트도 반드시 수집한다** — 다른 리뷰어가 이미 무엇을 지적했는지 모르고 쓴 리뷰는
중복 지적으로 신뢰를 잃는다:

```bash
# 대화 코멘트 + 리뷰 상태(승인/변경요청). body는 1500자로 잘라 토큰 낭비를 막는다
RTK_SKIP=1 gh pr view <번호|URL> --json comments,reviews \
  --jq '{comments: [.comments[] | {author: .author.login, body: .body[0:1500]}], reviews: [.reviews[] | {author: .author.login, state, body: .body[0:1500]}]}'

# 코드 라인에 달린 인라인 리뷰 코멘트 (in_reply_to_id가 같은 것끼리 한 스레드)
RTK_SKIP=1 gh api repos/<owner>/<repo>/pulls/<번호>/comments --paginate \
  --jq '[.[] | {id, author: .user.login, path, line, in_reply_to_id, body: .body[0:1500]}]'
```

봇 코멘트(`vercel`, `coderabbitai[bot]`, `github-actions` 등)는 사람 코멘트와 구분한다 —
봇 지적 중 타당한 것은 참고하되, 사람 리뷰어의 코멘트가 리뷰 맥락의 중심이다.
봇의 장문 코멘트는 잘려도 무방하지만, **사람** 코멘트가 잘렸으면 그 코멘트만 따로 전체를 읽는다.

## 3단계: 리뷰

다음 관점으로 본다. context.md의 중점 사항과 메모리에서 읽은 컨벤션 위반은 가장 먼저 체크한다.

1. **정확성** — 버그, 엣지 케이스, 잘못된 로직, 동시성/비동기 문제
2. **프로젝트 컨벤션** — conventions.md의 패턴 + 코드베이스의 기존 스타일과의 일관성
3. **설계** — 변경이 기존 구조와 어울리는지, 더 단순한 방법이 있는지
4. **테스트** — 변경에 대한 테스트가 있는지, 깨지기 쉬운 부분이 커버되는지
5. **보안** — 입력 검증, 인증/인가, 민감 정보 노출

심각도가 낮은 스타일 지적을 나열하는 것보다, 확신 있는 소수의 발견이 더 가치 있다.
불확실한 것은 단정하지 말고 질문으로 분류한다.

**기존 코멘트와의 교차 검토** — 2단계에서 수집한 코멘트를 발견 사항과 대조한다:

- 다른 리뷰어가 이미 지적한 것과 같은 발견은 새 발견처럼 쓰지 않는다. 동의하면 해당
  스레드에 근거를 보강하는 **답글**을 제안하고, 다르게 보면 반박 근거와 함께 답글로 분류한다.
- 기존 코멘트에 작성자가 이미 답했거나 후속 커밋으로 수정했으면 재지적하지 않는다
  (코멘트의 라인이 최신 diff에서 어떻게 바뀌었는지 확인).
- 아직 답이 없는 기존 지적 중 이번 변경에 실제로 중요해 보이는 것이 있으면 결과에서 짚어준다.

## 4단계: 결과 출력

리뷰를 읽는 사람이 (1) 문제가 된 **실제 코드**를 바로 보고, (2) **클릭 한 번**으로 GitHub의 해당 위치로 이동하고, (3) 동의하면 **코멘트를 복사해서 바로 달 수 있어야** 한다. 그래서 발견 사항은 한 줄 나열이 아니라 항목별 블록으로 쓴다. ALWAYS 이 구조를 사용한다:

````markdown
# PR #<번호> 리뷰: <제목>

## 요약
(이 PR이 무엇을 하는지 + 전체적인 평가 2~3문장)

## 발견 사항

### 🔴 반드시 수정 (머지 전 해결 필요)

#### 1. <문제를 한 줄로 요약한 제목>
📍 [`파일경로:시작-끝`](https://github.com/<owner>/<repo>/blob/<headRefOid>/<파일경로>#L<시작>-L<끝>)

```<언어>
<해당 코드 영역 — 문제가 보이는 최소한의 줄들>
```

**문제**: 왜 문제인지 1~3문장.
**제안**: 어떻게 고칠지. 가능하면 수정 코드를 보여준다.

**추천 코멘트** (복사해서 GitHub에 붙여넣기):
> 리뷰어가 실제로 달 만한 톤의 문구. 한 줄 수정으로 끝나는 경우
> ```suggestion 블록을 포함하면 작성자가 원클릭으로 반영할 수 있다.

### 🟡 권장 (고치면 좋음)
(같은 블록 구조, 추천 코멘트 포함)

### 💬 질문 / 확인 필요
(같은 블록 구조 — 추천 코멘트는 질문 형태로)

### 🔁 기존 코멘트 관련 (있는 경우만)

#### 1. <리뷰어>의 지적: <한 줄 요약>
📍 해당 스레드 위치 + 코멘트 링크

**상황**: 누가 무엇을 지적했고 현재 어떤 상태인지 1~2문장.
**내 판단**: 동의/이견과 그 근거.

**추천 답글** (해당 스레드에):
> 동의면 근거를 보강하는 답글, 이견이면 반박 근거를 담은 답글.

## 잘된 점
(있다면 1~2개. 억지로 만들지 않는다)
````

- 코드 블록은 diff 전체가 아니라 **문제가 보이는 최소 범위**만. 맥락이 필요하면 주변 2~3줄까지.
- 바로가기 링크는 반드시 `headRefOid` 고정 SHA를 쓴다 — 브랜치명 링크는 푸시되면 라인이 어긋난다.
- 추천 코멘트는 그대로 복사해 쓸 수 있는 완성문으로 쓴다. 기계적인 지적 톤이 아니라 동료에게 말하는 톤으로.

GitHub에 코멘트를 **게시**하는 것은 사용자가 명시적으로 요청할 때만 한다(5단계). 기본 동작은 위 결과(추천 코멘트 포함)를 대화에 보여주는 것이다.

## 5단계: 코멘트 게시 (사용자가 요청할 때만)

사용자가 "코멘트 달아줘", "2번만 달아줘" 처럼 게시를 요청하면:

1. **게시 전 확인** — 게시할 코멘트의 최종 본문과 위치를 보여주고 사용자 확인을 받는다.
   (사용자가 "전부 달아줘"라고 했고 본문이 이미 4단계 결과에 그대로 있다면 생략 가능)
2. 본문은 따옴표/개행 깨짐을 피하기 위해 임시 파일로 작성해서 전달한다.

```bash
# PR 전체에 대한 일반 코멘트
RTK_SKIP=1 gh pr comment <번호|URL> --body-file /tmp/comment.md

# 특정 코드 라인에 인라인 코멘트 (commit_id는 2단계의 headRefOid)
RTK_SKIP=1 gh api repos/<owner>/<repo>/pulls/<번호>/comments \
  -F body=@/tmp/comment.md -f commit_id=<headRefOid> \
  -f path='<파일경로>' -F line=<끝라인> -f side=RIGHT
# 여러 줄 범위면 -F start_line=<시작라인> -f start_side=RIGHT 추가

# 기존 스레드에 답글 (comment_id는 2단계에서 수집한 인라인 코멘트의 id)
RTK_SKIP=1 gh api repos/<owner>/<repo>/pulls/<번호>/comments/<comment_id>/replies \
  -F body=@/tmp/comment.md
```

- 인라인 코멘트의 `line`은 **diff에 포함된 라인**이어야 한다(변경되지 않은 라인에는 달 수 없음). 실패하면 가장 가까운 변경 라인으로 옮기거나 일반 코멘트로 대체한다.
- 게시 후 생성된 코멘트 URL을 사용자에게 보여준다.
- 승인/변경요청 같은 **리뷰 판정**(`gh pr review --approve/--request-changes`)은 사용자가 그 단어로 직접 요청할 때만 한다.

## 6단계: 메모리 저장

리뷰를 보여준 뒤, 결과를 기록한다:

1. `.review-memory/reviews/<YYYY-MM-DD>-pr-<번호>.md` 에 저장 — 위 리뷰 결과에 PR 제목/작성자/날짜를 헤더로 붙인 것. 다음 리뷰 때 맥락으로 쓸 수 있게 간결하게.
2. **conventions.md 업데이트 판단** — 이번 발견 중 "이 프로젝트에서 또 나올 법한 것"이 있는지 본다:
   - 과거 리뷰에서도 나온 이슈가 또 나왔다 → 해당 항목에 발생 횟수/날짜 추가
   - 새로운 프로젝트 고유 패턴을 발견했다 → 새 항목 추가 (무엇이 문제이고, 왜, 어떻게 피하는지 한 줄씩)
   - 일회성 실수는 기록하지 않는다. conventions.md가 잡동사니가 되면 읽는 의미가 없어진다.

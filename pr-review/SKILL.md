# PR 리뷰

GitHub PR을 리뷰한다. 핵심 차별점은 **프로젝트 메모리**다: 리뷰 전에 이 프로젝트에서
과거에 발견된 패턴을 읽고, 리뷰 후에 새로 알게 된 것을 기록한다. 리뷰를 거듭할수록
이 프로젝트에 특화된 리뷰어가 되는 것이 목표다.

## 1단계: 메모리 준비 및 로드

```bash
RTK_SKIP=1 bash ~/.claude/skills/review-memory/scripts/setup_memory.sh <프로젝트경로>
```

그 다음 반드시 읽는다:
- `.review-memory/conventions.md` — 이 프로젝트의 반복 패턴/컨벤션. 리뷰 관점에 반영한다.
- `.review-memory/reviews/` 의 최근 2~3개 파일 — 최근 리뷰 맥락. 같은 작성자/같은 영역이면 특히 유용하다.

메모리가 비어 있으면(첫 리뷰) 그냥 진행하면 된다.

> 환경 주의: 이 머신에서는 rtk 훅이 셸 명령을 재작성한다. 이 스킬의 모든 셸 명령
> (git/gh/스크립트)에는 `RTK_SKIP=1` 접두어를 붙여 재작성을 우회한다 — 훅 래퍼가
> 이 마커를 인식해 원본 명령을 그대로 실행한다. 그래도 `_safe_eval` 오류가 나면
> `/usr/bin/git` 절대경로로 재시도하고, 파일 읽기/쓰기는 Read/Write 도구를 사용한다.

## 2단계: PR 정보 수집

```bash
RTK_SKIP=1 gh pr view <번호|URL> --json title,body,author,baseRefName,headRefName,files
RTK_SKIP=1 gh pr diff <번호|URL>
```

- PR이 지정되지 않았으면 `gh pr list`로 열린 PR을 보여주고 어떤 것인지 확인한다. 현재 브랜치에 연결된 PR이 있으면 그것을 기본 후보로 제시한다.
- diff만으로 판단이 어려운 부분은 저장소에서 관련 파일을 직접 읽는다. 변경된 함수의 호출처, 수정된 타입의 사용처 등 **diff 바깥의 영향**을 확인하는 것이 좋은 리뷰와 표면적인 리뷰의 차이다.

## 3단계: 리뷰

다음 관점으로 본다. 메모리에서 읽은 컨벤션 위반은 가장 먼저 체크한다.

1. **정확성** — 버그, 엣지 케이스, 잘못된 로직, 동시성/비동기 문제
2. **프로젝트 컨벤션** — conventions.md의 패턴 + 코드베이스의 기존 스타일과의 일관성
3. **설계** — 변경이 기존 구조와 어울리는지, 더 단순한 방법이 있는지
4. **테스트** — 변경에 대한 테스트가 있는지, 깨지기 쉬운 부분이 커버되는지
5. **보안** — 입력 검증, 인증/인가, 민감 정보 노출

심각도가 낮은 스타일 지적을 나열하는 것보다, 확신 있는 소수의 발견이 더 가치 있다.
불확실한 것은 단정하지 말고 질문으로 분류한다.

## 4단계: 결과 출력

ALWAYS 이 구조를 사용한다:

```markdown
# PR #<번호> 리뷰: <제목>

## 요약
(이 PR이 무엇을 하는지 + 전체적인 평가 2~3문장)

## 발견 사항
### 🔴 반드시 수정 (머지 전 해결 필요)
- `파일:라인` — 문제와 이유, 가능하면 수정 제안

### 🟡 권장 (고치면 좋음)
- ...

### 💬 질문 / 확인 필요
- ...

## 잘된 점
(있다면 1~2개. 억지로 만들지 않는다)
```

GitHub에 코멘트를 다는 것은 사용자가 명시적으로 요청할 때만 한다. 기본 동작은 위 결과를 대화에 보여주는 것이다.

## 5단계: 메모리 저장

리뷰를 보여준 뒤, 결과를 기록한다:

1. `.review-memory/reviews/<오늘날짜>-pr-<번호>.md` 에 저장 — 위 리뷰 결과에 PR 제목/작성자/날짜를 헤더로 붙인 것. 다음 리뷰 때 맥락으로 쓸 수 있게 간결하게.
2. **conventions.md 업데이트 판단** — 이번 발견 중 "이 프로젝트에서 또 나올 법한 것"이 있는지 본다:
   - 과거 리뷰에서도 나온 이슈가 또 나왔다 → 해당 항목에 발생 횟수/날짜 추가
   - 새로운 프로젝트 고유 패턴을 발견했다 → 새 항목 추가 (무엇이 문제이고, 왜, 어떻게 피하는지 한 줄씩)
   - 일회성 실수는 기록하지 않는다. conventions.md가 잡동사니가 되면 읽는 의미가 없어진다.
